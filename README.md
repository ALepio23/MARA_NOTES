# MARA.notes — viewer

Personal note hub for Alessandro Alvigi. Markdown files in this directory render in a static viewer (`index.html`) deployable on GitHub Pages. Content is encrypted client-side (AES-GCM + PBKDF2) — the published URL shows a password gate; without the password, nothing is readable.

## How it works

- `*.md` files in this directory are the source of truth (one doc each, YAML frontmatter + markdown body).
- `index.json` is an aggregate index regenerated by `sync.py`.
- `index.html` is the static viewer. The doc payload is **encrypted** with the password stored at `~/.config/mara-notes-pass` (never committed). The browser decrypts it client-side using the Web Crypto API.
- `_inbox/` and `_archive/` are scratch folders for the `mara-notes-architect` Claude plugin workflow.

## Workflow

After adding or editing any `.md` file:

```bash
python3 sync.py
git add .
git commit -m "update notes"
git push
```

GitHub Pages auto-deploys on push to `main`. Site updates in ~30s.

## Password

Stored locally at `~/.config/mara-notes-pass`. Never committed, never logged. To rotate:

```bash
printf 'new-password' > ~/.config/mara-notes-pass
chmod 600 ~/.config/mara-notes-pass
python3 sync.py
git add index.html index.json
git commit -m "rotate viewer password"
git push
```

Old password no longer works after sync — anyone who had the URL needs the new password.

## Security model

- **What's protected**: the JSON body of all notes (titles, content, people, companies, projects). Encrypted with AES-GCM-256, key derived via PBKDF2-HMAC-SHA256 with 200,000 iterations and a random 16-byte salt.
- **What's exposed**: the viewer skeleton (HTML structure, CSS, JS code), the existence of the repo, the URL. Without the password, the encrypted blob is indistinguishable from random bytes.
- **Brute-force resistance**: 200K PBKDF2 iterations → ~200ms per attempt on modern hardware. A weak password (dictionary word, short) is still brute-forceable. A long passphrase is safe.
- **Not a substitute for**: hosting on a private network for highly sensitive content. This is reasonable security for "internal team / advisor" content, not for credentials, financials, or legally privileged material.

## Deploy (first time)

1. Create a **private** repo on github.com (no description, no README — they're already here).
2. Add remote:
   ```bash
   git remote add origin git@github.com:<user>/<repo>.git
   git push -u origin main
   ```
3. On the repo page: Settings → Pages → Source: Deploy from a branch → Branch: `main` → `/ (root)` → Save.
4. Wait ~1 min for first deploy. URL appears at the top of the Pages settings.
